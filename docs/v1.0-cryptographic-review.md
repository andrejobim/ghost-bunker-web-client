## Client-specific scope for future audit

Evaluators should review:

- **Room key generation** — `crypto.getRandomValues`, 32-byte import
- **AES-256-GCM** — nonce length, AAD construction, `cipher_suite` wire label vs actual keying
- **Invite link** — `#gbkey=` in URL fragment; fragment cleanup; query vs fragment leakage
- **Storage policy** — no `room_key` in `localStorage` by default (`docs/e2ee-v0.2.md`)
- **Logging** — no secrets in console or persisted logs

---

## v0.2 experimental model (unchanged)

- Wire protocol remains **v0.1** (`GhostEnvelope`, subprotocol `ghost-bunker.v0.1`).
- E2EE is **symmetric room key + AES-GCM** — experimental, not audited.
- Do **not** implement X25519, Ed25519, Double Ratchet, or multi-device sync in v0.x prompts.

---

## Compatibility rules (client)

1. Do not change encryption semantics without a matching server release and test matrix
   (server `docs/protocol-migration-plan.md`).
2. Do not change Protobuf definitions here; `.proto` lives in the server repo.
3. Run `npm test` before any release that touches `src/crypto/` or envelope encoding.
